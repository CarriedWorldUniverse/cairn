# cairn — bisect

**Status:** draft for approval · 2026-06-24
**Goal:** `cairn bisect` — a guided binary search to find the commit that introduced a regression, over a line's sealed history. Manual (`good`/`bad`/`skip`) **and** automated (`cairn bisect run <cmd>`). Discrete CLI (CLI + agent-first — `run` lets an agent hunt a regression autonomously).

**Operator decision (locked):** include `bisect run`.

---

## 1. Model & the critical WCC interaction

Bisect searches the **sealed chain** of a line between a known-`good` commit and a known-`bad` commit (good is an ancestor of bad). Candidates = the sealed commits in `(good, bad]`; the answer is the **first bad** commit. At each step it **materializes the midpoint commit into the line's expressed folder** so you can build/test it.

**Critical interaction with working-copy-is-a-commit:** while a bisect session is active, the expressed folder shows a *historical* commit, not the working change. So the command-start auto-snapshot (`SyncWorking`) **must be suspended for the whole session** — otherwise it would capture the historical midpoint into your working change, corrupting it. Implementation: `openRepoSynced` (and any auto-snapshot path) checks `Repo.BisectActive()` and **skips `SyncWorking`** while a session exists. `cairn bisect reset` re-materializes the working tip and clears the session.

**Preconditions:** `start` requires a **clean working change** (no un-sealed delta) — refuse otherwise with "stash or commit your work before bisecting". The session records `restore_tip` (the line tip at start) so `reset` always restores it.

## 2. State (`bisect` table — single active session)
```sql
CREATE TABLE IF NOT EXISTS bisect (
  id          INTEGER PRIMARY KEY CHECK (id = 1),  -- one active session per repo
  line_id     TEXT NOT NULL,
  branch      TEXT NOT NULL,
  good_sha    TEXT NOT NULL,   -- current known-good lower bound (narrows on 'good')
  bad_sha     TEXT NOT NULL,   -- current known-bad upper bound (narrows on 'bad')
  current_sha TEXT NOT NULL,   -- the midpoint currently materialized for testing
  restore_tip TEXT NOT NULL,   -- line tip at start, restored on reset
  started_at  TEXT NOT NULL
);
```
(Additive `CREATE TABLE IF NOT EXISTS` — existing repos get it on Open. `BisectActive()` = a row exists.)

## 3. Algorithm (engine, `internal/change/bisect.go`)
Operate on `chain := sealedChain(lineID)` (base→top, from history-editing). Let `gi = index(good)`, `bi = index(bad)` in `chain` (require `gi < bi`, both present → else "good is not an ancestor of bad on this line").
- **midpoint(gi, bi):** the candidates are `chain[gi+1 .. bi]`; midpoint index `mi = gi + (bi-gi)/2` (rounding so it's a real candidate strictly between when possible). The commit to test = `chain[mi].Commit`.
- **`BisectStart(branch, good, bad) (current string, done bool, firstBad string, err error)`:** validate; require clean working change; INSERT the session `{good, bad, restore_tip = line.TipCommit}`; if `bi == gi+1` (bad is the immediate next sealed commit after good) → **done**, `firstBad = bad`; else `current = midpoint`, store it, return it.
- **`BisectMark(verdict) (current string, done bool, firstBad string, err error)`** where verdict ∈ {good, bad}:
  - `good` → the regression is AFTER current: set `good_sha = current_sha` (narrow lower bound up).
  - `bad` → the regression is AT or BEFORE current: set `bad_sha = current_sha`.
  - recompute `gi/bi`; if `bi == gi+1` → **done**, `firstBad = bad_sha`; else recompute midpoint, store `current`, return it.
- **`BisectSkip() (current string, err error)`** (basic): pick the candidate adjacent to the current midpoint (one step toward `good`) as the new `current` without moving the bounds; if no untested candidate remains, return "cannot narrow further — remaining commits skipped". (v1 keeps skip simple; full skip-set tracking is deferred.)
- **`BisectReset() (restoreTip string, err error)`:** read `restore_tip`, DELETE the session, return the tip (caller re-materializes).
- **`BisectStatus() (BisectInfo, error)`:** the session + remaining candidate count (`bi - gi`).
All single-row upserts/updates; no working-change writes.

## 4. Worktree + CLI
- `Repo.BisectActive() (bool, error)`; **`openRepoSynced` skips `SyncWorking` when active** (the key change). 
- `Repo.BisectStart(branch, good, bad)` → engine + **materialize `current` into the expressed folder** (so you test it) + return the step (or the done/first-bad result, in which case re-materialize the restore tip). Refuse if the working change is dirty.
- `Repo.BisectMark(verdict)` → engine + materialize the new `current` (or, on done, leave the first-bad materialized and tell the user to `reset`).
- `Repo.BisectReset()` → engine returns `restore_tip` → re-materialize the working tip into the folder → clear.
- `Repo.BisectStatus()`.
- **CLI** (`cmd/cairn/main.go`), all routed through `cmdBisect` sub-dispatch:
  - `cairn bisect start --bad <commit> --good <commit> [branch]` — begin; prints the midpoint to test (`testing <sha> — N candidates left`) to stderr; the folder now holds that commit.
  - `cairn bisect good` / `cairn bisect bad` — mark the current midpoint; prints the next midpoint, or `first bad commit: <sha>` when done.
  - `cairn bisect skip` — skip the current midpoint.
  - `cairn bisect reset` — end the session, restore your working folder.
  - `cairn bisect status` — show the session (good/bad bounds, current, candidates left).
  - `cairn bisect run <cmd...>` — automate (§5).
  These commands open the repo but (while active) do NOT auto-snapshot. `start` uses the normal sync path BEFORE the session exists to capture/clean-check the working state.

## 5. `cairn bisect run <cmd...>`
After `start` (or as `bisect run` following a start), loop:
1. The current midpoint is materialized in the folder.
2. Exec `<cmd>` with `cmd.Dir = <expressed folder>`. Map the exit code: **0 → good**, **125 → skip**, **non-zero (1–124, 126+) → bad** (git's convention; v1: 0=good, 125=skip, else=bad).
3. `BisectMark`/`Skip` accordingly; materialize the next midpoint; repeat until **done**.
4. Print `first bad commit: <sha>` and its subject; leave the session (user runs `cairn bisect reset` to restore). Stream each step's verdict to stderr.
`run` requires an active session (from a prior `start`) OR accepts `--bad/--good` to start-and-run in one shot — pin: `cairn bisect run <cmd>` requires a prior `start` (keep it simple; start-and-run is a later convenience).

## 6. Out of scope (later)
Full `skip` set tracking + "first bad is one of N" reporting; `bisect log`/`replay`; bisecting across lines/merges; `bisect run` start-in-one-shot; visualize. The pre-existing WCC follow-ups remain.

## 7. Testing / DoD
- **manual**: a line with N sealed commits where commit K introduces a regression (e.g. `flag.txt` flips from "ok" to "bad" at K). `bisect start --good <S1> --bad <Sn>` → materializes a midpoint; mark `good`/`bad` per whether the midpoint's `flag.txt` is "ok"/"bad" → converges to K as "first bad commit"; `reset` restores the working tip.
- **suspend-sync**: while bisecting, running another command (e.g. `status`) does NOT snapshot the historical midpoint into the working change (the working change's head is unchanged across bisect steps); after `reset`, the working change + folder are intact.
- **clean guard**: `start` with an un-sealed working delta → refused.
- **run**: `bisect run sh -c 'grep -q ok flag.txt'` (exit 0 when good) drives to the first-bad commit automatically.
- **status/skip**: status shows bounds + candidates; skip moves off the current midpoint.
- Full gate + cross-compile; `skipOnWindows` on e2e (esp. `run` with a shell); all prior phases unaffected. Atomic session writes.
